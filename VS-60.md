# Plan: Bidirectional LSP Navigation for Soul Markdown Docs

## Context

The `soul-lsp` server currently provides hover, goto-definition, and find-references only when the cursor is on a code annotation. All three handlers call `annotation_at()` and return `None` immediately if the cursor is in any other file — including Soul markdown docs. This means opening a `.docs/**/*.md` Soul document gives no LSP intelligence at all: you can't jump to the annotated code, see what implements it, or hover for a summary.

The fix adds a `document_at()` fallback so all three handlers work bidirectionally — from doc → code as well as code → doc.

---

## Critical File

- `crates/soul-lsp/src/server.rs` — only file modified

## Implementation

### Step 1 — Add `document_at` helper

Insert after `linked_doc` (line 57), before `to_uri`:

```rust
fn document_at<'g>(graph: &'g SemanticGraph, root: &Path, uri: &Uri) -> Option<&'g Document> {
    let req_path = uri.to_file_path()?;
    let req_path = req_path.canonicalize().unwrap_or(req_path.to_path_buf());
    graph.documents.iter().find(|d| {
        let abs = if d.path.is_absolute() {
            d.path.clone()
        } else {
            root.join(&d.path)
        };
        let canon = abs.canonicalize().unwrap_or(abs);
        canon == req_path
    })
}
```

Same shape as `annotation_at` — resolves path, ignores line/character (any position in the doc triggers it).

---

### Step 2 — Update `hover`

Replace the early-return `else` branch with a doc fallback:

```rust
// BEFORE
let Some(ann) = annotation_at(...) else {
    return Ok(None);
};
let mut md = format!("**{}**", ann.id);
if let Some(doc) = linked_doc(graph, &ann.id) {
    if let Some(title) = &doc.title {
        md.push_str(&format!("\n\n*{title}*"));
    }
    md.push_str(&format!("\n\n`{}`", doc.path.display()));
}

// AFTER
let (id, md) = if let Some(ann) = annotation_at(graph, &self.root, &pos.text_document.uri, pos.position.line) {
    let mut md = format!("**{}**", ann.id);
    if let Some(doc) = linked_doc(graph, &ann.id) {
        if let Some(title) = &doc.title {
            md.push_str(&format!("\n\n*{title}*"));
        }
        md.push_str(&format!("\n\n`{}`", doc.path.display()));
    }
    (ann.id.clone(), md)
} else if let Some(doc) = document_at(graph, &self.root, &pos.text_document.uri) {
    let mut md = format!("**{}**", doc.id);
    if let Some(title) = &doc.title {
        md.push_str(&format!("\n\n*{title}*"));
    }
    let locs: Vec<_> = graph.annotations.iter().filter(|a| a.id == doc.id).collect();
    if !locs.is_empty() {
        md.push_str(&format!("\n\n{} code location(s):", locs.len()));
        for a in &locs {
            md.push_str(&format!("\n- `{}:{}`", a.path.display(), a.line));
        }
    }
    (doc.id.clone(), md)
} else {
    return Ok(None);
};
```

Result: hovering anywhere in a Soul doc shows id, title, and a list of annotated files+lines.

---

### Step 3 — `goto_definition` — no change

The Soul doc IS the definition. Annotations are references to it. `goto_definition` from a markdown file correctly returns `None` — no fallback needed.

---

### Step 4 — Update `references`

Same pattern — add doc fallback after the `annotation_at` branch:

```rust
// BEFORE
let Some(ann) = annotation_at(...) else {
    return Ok(None);
};
let id = ann.id.clone();
// → filters annotations by id

// AFTER
let id = if let Some(ann) = annotation_at(graph, &self.root, &pos.text_document.uri, pos.position.line) {
    ann.id.clone()
} else if let Some(doc) = document_at(graph, &self.root, &pos.text_document.uri) {
    doc.id.clone()
} else {
    return Ok(None);
};
// existing filter+collect unchanged below
```

Result: find-references from a Soul doc lists all code locations — same result as goto-definition but via the references UI.

## Semantic Classification

Soul's model has a clean definition/reference split:

| Concept | Role |
|---|---|
| Soul document (`.md`) | **Definition** — the authoritative specification. Exactly one per ID. |
| Code annotation (`#[soul(...)]`, `[Soul(...)]`) | **Reference** — a claim that a piece of code implements or participates in that concept. Many per ID, across any language. |

This is already implicitly correct in the existing LSP code (`goto_definition` from an annotation → doc; `references` from an annotation → all annotations). The plan makes it explicit and symmetric.

## Step 5 — Update MCP tool descriptions (optional but recommended)

`soul_explain` is the only MCP tool that surfaces both documents and annotations together. Its current description never uses the words *definition* or *reference*, so the semantic relationship is implicit. Two targeted edits to `crates/indexer/src/mcp/mod.rs`:

**Documents section** (currently: `"Documents: metadata only — kind…"`)
Add the classification:
> "Documents: the **definition** — the authoritative specification for this ID. At most one per ID. Returns metadata only (kind, title, file path); follow the Read prompt to get the full specification body."

**Annotations section** (currently: `"Annotations: every location in the codebase tagged with this ID…"`)
Add the classification:
> "Annotations: the **references** — every location in the codebase that implements or participates in this concept. Many per ID across any language or layer."

`soul_index`'s overview description introduces the model and is already clear enough — no change needed there.
