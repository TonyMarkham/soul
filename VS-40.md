# Plugin SDK Extraction + `role` Cleanup

## Context

Soul's annotation parsing is currently hardcoded in `crates/indexer/` with two built-in parsers: `RustParser` (`annotation/rust.rs`) and `CSharpParser` (`annotation/c_sharp.rs`). Dispatch is a static array at `annotation/mod.rs:22`. There is no extension point — a new language means editing core indexer code.

Tony wants to dogfood the Soul SDK by extracting these parsers into first-party plugin crates behind a proper SDK trait. The indexer consumes language support as plugins rather than hardcoding it.

A second problem is tangled into this: the indexer has drifted from the ground-truth attribute crates. `crates/soul-attributes` (Rust `#[soul(...)]` macro) and `packages/Soul.Attributes` (C# `[Soul(...)]` attribute) both define annotations as **`id` + opaque metadata bag, nothing privileged**. But the indexer invented a first-class `role: Option<String>` field and baked it into:

- `NormalizedAnnotation` and `CodeAnnotation` structs
- SQLite `annotations.role` column
- MCP tool descriptions (literally shows `role = "backend"` / `Role = "frontend"` in syntax examples)
- `mcp/format.rs` output (prints `[role=backend]`)
- Test fixtures, scan tests, graph tests, explain tests
- Self-referential `#[soul(id = "...", role = "implementation")]` on the indexer's own functions

Every session, Claude reads this code and faithfully reproduces the fictional `role` concept. The plugin SDK refactor is the natural moment to rip it out: parsers move into plugin crates that never emit `role`, the indexer-side types drop the field, and the privileged concept dies.

Also in scope: a new `soul_annotation_syntax(target)` MCP tool. When `soul_list_gaps` finds an unlinked file (e.g. `foo.go`) and the LLM needs to add an annotation, it calls this tool with the file path or extension. The indexer routes to the plugin registered for that extension, which returns its syntax guidance. Core stays language-agnostic — it never knows Rust or C# syntax, just dispatches to plugins.

**Plugins are dynamically loaded shared libraries listed in `soul.toml`.** Each plugin is a `cdylib` (`.dylib` / `.so` / `.dll`) built from its own crate and dropped into a path the indexer can `dlopen` at startup. `soul.toml` contains an `[[plugins]]` array where each entry is `{ language = "rust", path = "./.soul/plugins/rust.dylib" }`. The indexer depends on the SDK only — the plugin crates are NOT compile-time dependencies of the indexer. This means:

- Adding a new plugin (Go, Python, TypeScript) requires **zero indexer code changes** — just drop a new cdylib into the plugins directory and add an `[[plugins]]` entry.
- Plugins can come from anywhere: first-party, third-party, a developer's own scratch plugin. All go through the same loader path.
- The indexer binary is small and knows nothing about Rust or C# annotation syntax. That knowledge lives exclusively in the plugin shared libraries.

**FFI approach.** Rust `dyn Trait` has no stable ABI, so the plugin boundary uses the [`abi_stable`](https://docs.rs/abi_stable) crate. The SDK defines the parser trait with `#[sabi_trait]`, which generates FFI-safe wrapper types. Plugins export a root module via `#[sabi_extern_fn]`. The loader uses `abi_stable`'s `RootModule::load_from_file` to dlopen each path, pull out the parser instance, and store it in the runtime registry. Battle-tested, stable ABI guaranteed across abi_stable versions.

The current `[scan] annotation_extensions` field — which today implicitly selects "Rust and C# files are scanned" — is removed. Its role is taken over by plugin loading: each loaded plugin declares its own extension via the SDK trait's `extension()` method, and the indexer builds the extension→plugin routing table at startup from the loaded set. Single source of truth.

**Intended outcome**

- Three new crates: `soul-plugin-sdk` (rlib), `soul-plugin-rust` (cdylib), `soul-plugin-csharp` (cdylib).
- Indexer depends on `soul-plugin-sdk` + `abi_stable` only. No compile-time dependency on the plugin crates.
- `role` is completely gone from the indexer — types, schema, format, MCP descriptions, fixtures, tests, self-annotations.
- `soul.toml` gains an `[[plugins]]` array. `annotation_extensions` is removed.
- New `soul_annotation_syntax` MCP tool routes extension → loaded plugin → guidance string.
- Adding a third-party language plugin is a pure drop-in operation: build a cdylib against `soul-plugin-sdk`, ship it, add an `[[plugins]]` entry. No indexer rebuild.

### SDK trait (new, abi_stable)

Lives in `crates/soul-plugin-sdk/src/parser.rs`. The trait is declared with `#[sabi_trait]`, which the macro expands into an FFI-safe wrapper type:

```rust
use abi_stable::{sabi_trait, std_types::{RString, ROption, RResult}};

#[sabi_trait]
pub trait AnnotationParser: Send + Sync {
    /// File extension this plugin handles (e.g. "rs", "cs", "go"). No dot.
    fn extension(&self) -> RString;

    /// Short stable identifier for the annotation syntax (e.g. "rust-attribute",
    /// "csharp-attribute"). Stored in the SQLite `syntax` column.
    fn syntax(&self) -> RString;

    /// Parse a single source line. Returns ROption::RNone if the line isn't an
    /// annotation, RSome(RErr) on malformed input, RSome(ROk) on success.
    fn parse_line(&self, line: RStr<'_>) -> ROption<RResult<NormalizedAnnotation, AnnotationError>>;

    /// Human-readable guidance shown to LLMs via the `soul_annotation_syntax`
    /// MCP tool. Should include syntax template, placement rule, example.
    fn syntax_guidance(&self) -> RString;
}
```

The macro generates `AnnotationParser_TO` (the FFI-safe trait object type) which is what the loader and the indexer actually hold.

### SDK root module (new)

Lives in `crates/soul-plugin-sdk/src/root_module.rs`. `abi_stable` requires each plugin to expose a root module struct. The SDK defines the shape; plugins implement it:

```rust
use abi_stable::{StableAbi, library::RootModule, package_version_strings, sabi_types::VersionStrings};

#[repr(C)]
#[derive(StableAbi)]
#[sabi(kind(Prefix(prefix_ref = "SoulPluginRef")))]
#[sabi(missing_field(panic))]
pub struct SoulPlugin {
    /// Returns the parser instance.
    pub parser: extern "C" fn() -> AnnotationParser_TO<'static, RBox<()>>,
}

impl RootModule for SoulPluginRef {
    abi_stable::declare_root_module_statics! {SoulPluginRef}
    const BASE_NAME: &'static str = "soul_plugin";
    const NAME: &'static str = "soul_plugin";
    const VERSION_STRINGS: VersionStrings = package_version_strings!();
}
```

Plugins instantiate this struct and export it via `#[export_root_module]`. The loader calls `SoulPluginRef::load_from_file(&path)` which dlopens the library, looks up the exported symbol, and hands back a typed reference.

### SDK types (new)

`crates/soul-plugin-sdk/src/normalized_annotation.rs` — **role is GONE**, uses FFI-safe types:

```rust
#[repr(C)]
#[derive(StableAbi, Debug, Clone)]
pub struct NormalizedAnnotation {
    pub id: RString,
    pub metadata: RHashMap<RString, RString>,
    pub raw: RString,
}
```

Note: metadata was `serde_json::Map<String, Value>` in the current code, nesting arbitrary JSON. Across the FFI boundary, `abi_stable` has no direct JSON Value type. Options: (a) flatten to `RHashMap<RString, RString>` and keep values as JSON strings if they're objects, (b) ship a vendored JSON value type in the SDK. Recommendation: (a) for MVP — simpler, matches what the parsers actually produce today (string values for almost everything; the one exception is the C# `MetadataJson=` field which already carries pre-serialized JSON). The indexer converts back to `serde_json::Value` on the way into `CodeAnnotation` if needed. Document this in the SDK README so plugin authors know the contract.

`crates/soul-plugin-sdk/src/error.rs` — the two existing error variants (`Malformed`, `MissingId`) moved out of `crates/indexer/src/error/annotation.rs`, rewritten as an FFI-safe enum. `ErrorLocation` isn't FFI-safe today, so the SDK error carries `RString` for the message and the indexer wraps it back into the local `IndexerError` (with a fresh `ErrorLocation` captured at the wrap site) when turning SDK errors into diagnostics. This preserves the IndexerError style memory for core errors while keeping the plugin boundary clean.

Notes on the new shape:
- `[[plugins]]` is an array of tables. Each entry is a plugin the indexer should dlopen at startup.
- `language` is a friendly identifier used in error messages and to detect duplicate entries. It does NOT need to match any compile-time enum on the indexer side; it's just a string.
- `path` is a filesystem path to the plugin shared library. Resolved relative to the repo root (the directory containing `.soul/`). Absolute paths also accepted.
- The path is platform-specific — `.dylib` on macOS, `.so` on Linux, `.dll` on Windows. A team working across platforms will need to check in platform-conditional configs or pick a convention (e.g. `.soul/plugins-macos/rust.dylib`, `.soul/plugins-linux/rust.so`). This plan does not attempt to solve cross-platform path resolution; it's listed in Out of Scope.
- `annotation_extensions` is deleted. Each loaded plugin declares its extension via the SDK trait's `extension()` method. The indexer builds the extension→plugin map at startup from the loaded set.
- Duplicate `language` values or two plugins claiming the same extension are hard errors at config load time.
- `soul.toml.example` at the repo root is updated to match. The repo's own `.soul/soul.toml` is updated.

### Plugin loader (replaces registry)

New file `crates/indexer/src/annotation/loader.rs`. The loader opens each configured plugin shared library, holds the opened library handles for the process lifetime, and exposes the loaded parsers as an iterable registry.

```rust
use abi_stable::library::RootModule;
use soul_plugin_sdk::{AnnotationParser_TO, SoulPluginRef};

pub struct LoadedPlugin {
    pub language: String,
    pub parser: AnnotationParser_TO<'static, RBox<()>>,
}

pub struct PluginRegistry {
    plugins: Vec<LoadedPlugin>,
}

impl PluginRegistry {
    pub fn load(entries: &[PluginEntry], root: &Path) -> IndexerResult<Self> {
        let mut plugins = Vec::with_capacity(entries.len());
        for entry in entries {
            let path = if entry.path.is_absolute() {
                entry.path.clone()
            } else {
                root.join(&entry.path)
            };
            let module = SoulPluginRef::load_from_file(&path)
                .map_err(|e| IndexerError::plugin_load(entry.language.clone(), path.clone(), e))?;
            let parser = (module.parser())();
            plugins.push(LoadedPlugin { language: entry.language.clone(), parser });
        }
        validate_unique_languages(&plugins)?;
        validate_unique_extensions(&plugins)?;
        Ok(Self { plugins })
    }

    pub fn parser_for_extension(&self, ext: &str) -> Option<&AnnotationParser_TO<'static, RBox<()>>> { ... }
    pub fn iter(&self) -> impl Iterator<Item = &LoadedPlugin> { ... }
}
```

Lifetime note: `abi_stable`'s root module API holds the loaded library alive for the process lifetime once loaded, so returning `AnnotationParser_TO<'static, RBox<()>>` is sound. The `PluginRegistry` is built once at the start of `scan_repository` (and again in MCP tool handlers that don't go through `scan_repository`) and threaded through the annotation pipeline.

New `IndexerError` variants (per the IndexerError style memory — named fields, `ErrorLocation`, `#[track_caller]` constructor, each in its own file under `crates/indexer/src/error/`):
- `PluginLoadFailed { language: String, path: PathBuf, source: String, location: ErrorLocation }` — wraps `abi_stable`'s `LibraryError`
- `DuplicatePluginLanguage { language: String, location: ErrorLocation }`
- `DuplicatePluginExtension { extension: String, first: String, second: String, location: ErrorLocation }`

### Indexer ↔ plugin data conversion

The SDK trait returns `NormalizedAnnotation` built from `abi_stable` types (`RString`, `RHashMap`). The indexer's `CodeAnnotation` uses std types (`String`, `serde_json::Map`). A thin conversion layer in `crates/indexer/src/annotation/mod.rs` turns one into the other: walk the `RHashMap`, convert each key/value to `String`, and place into a `serde_json::Map<String, Value>`. Most values are strings, but the C# `MetadataJson=` path produces pre-serialized JSON — those entries are detected by attempting `serde_json::from_str` on the value and falling back to `Value::String` on parse failure. The same contract must be documented in the SDK so plugin authors know how to pack metadata.

### First-party plugin distribution + `indexer init`

Since the indexer no longer compiles the plugin code in, the first-party cdylibs must reach `./.soul/plugins/` somehow. Two things change:

1. **`indexer init`** — when initializing a new repo, `indexer init` creates the `.soul/plugins/` directory and copies the first-party plugin cdylibs into it. The indexer discovers source cdylibs by looking in a sibling `plugins/` directory next to its own executable path (resolved via `std::env::current_exe()`). When Tony runs `./.soul/indexer init`, the indexer looks for `./.soul/plugins-src/rust.dylib` (or equivalent), copies it to `./.soul/plugins/rust.dylib`, and seeds `soul.toml` with the corresponding `[[plugins]]` entry.

2. **Packaging convention for the dev workflow** — in Tony's current dev loop, the indexer is built by `cargo build -p indexer` and landing at `./target/release/indexer`, then copied/symlinked into `./.soul/indexer`. To keep `indexer init` working, the dev workflow script must also copy the built `libsoul_plugin_rust.dylib` and `libsoul_plugin_csharp.dylib` from `./target/release/` into `./.soul/plugins-src/` (or a similar staging directory) before running `indexer init`. This is a manual dev-only concern; the plan documents it but does not automate it beyond updating any existing build-and-install script in the repo if one exists.

For the self-dogfooding case (this repo), the existing `./.soul/indexer` binary and the existing `./.soul/soul.toml` are updated in-place as part of step 12 ("Manual verification") rather than going through `indexer init`.

Cross-platform file extensions: `indexer init` picks the correct suffix at runtime (`.dylib` on macOS, `.so` on Linux, `.dll` on Windows) based on `std::env::consts::DLL_SUFFIX`, and writes matching paths into the seeded `soul.toml`. Users committing `soul.toml` to git on multi-platform teams will need to work around this; see Out of Scope.

### Test infrastructure note

Integration tests in `crates/indexer/tests/` that exercise the annotation pipeline need real plugin cdylibs available at runtime. Cargo builds cdylibs into `target/{profile}/` or `target/{profile}/deps/` on a per-platform suffix. Options for test harness:

- Add `soul-plugin-rust` and `soul-plugin-csharp` as `[dev-dependencies]` of the indexer crate (cargo will ensure they're built before tests run; the cdylib artifacts will land in `target/{profile}/deps/`)
- At test startup, compute the target dir and build plugin paths relative to it — `env!("CARGO_TARGET_TMPDIR")` or parse `cargo metadata` output
- Write a `tests/common/mod.rs` helper that returns a `PluginEntry` list with resolved paths for use in every test

The cleanest path is dev-dependencies + a test helper that resolves `target/deps/libsoul_plugin_{rust,csharp}.{dylib|so|dll}` at test time. Unit tests inside the plugin crates themselves (`crates/soul-plugin-*/tests/parse.rs`) don't need this — they test the parsers directly as Rust types, not through FFI.

## Out of scope

- **Automated first-party plugin packaging.** `indexer init` does a best-effort copy from a sibling `plugins-src/` directory, but assembling a distributable `indexer` binary with its plugins bundled (installer, homebrew formula, tarball layout) is not in scope. Tony handles staging manually in the self-dogfooding case.
- **Cross-platform `soul.toml` path resolution.** Plugin paths are literal strings with platform-specific extensions. A team working across macOS + Linux + Windows will need a convention (e.g. platform-conditional `soul.toml` variants) or will need to accept that the committed `soul.toml` only works on one platform. Solving this properly (extension resolution, per-OS overrides) is a follow-up.
- **Third-party plugin distribution mechanism.** No plugin registry, no versioning, no signature verification. `abi_stable` provides an ABI-stability check at load time, but anything beyond that (reputation, sandboxing) is not in scope.
- **A plugin for any language other than Rust and C#.** Go, Python, TypeScript, etc. are follow-up plugins, proven by the drop-in test in Verification.
- **Changing the ground-truth attribute crates** (`crates/soul-attributes`, `packages/Soul.Attributes`) — they are already correct.
- **Renaming the existing `soul-attributes` crate.** The `-sdk` / `-plugin-*` names sit alongside it without conflict.
